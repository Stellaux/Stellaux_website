Future Scaling Suggestions

Where the current design will bend first, and the cheapest next move at each stage. Read top-to-bottom: each section assumes the previous ones are in place.

Premise: the v1 stack — one Rust API instance + managed Postgres + Cloudflare Workers storefront — should comfortably carry tens of thousands of orders/month without any of the changes below. Most of this page is "what to do when", not "what to do now". Resist building ahead of the load.

Growth stages & cheapest next move

StageSignalCheapest next move
S1 · 0–1K orders/mo Launch volume Stay on v1. One API instance, managed Postgres starter tier, Workers free plan.
S2 · 1K–10K orders/mo Single instance shows occasional p95 spikes during peaks Run 2 API instances behind the LB; bump Postgres to a small dedicated tier. No code changes.
S3 · 10K–50K orders/mo Webhook delivery occasionally times out; admin queries are slow Add a read replica for admin/BI reads; move webhook processing to a background queue. Add Redis for cart / rate limiting / locks.
S4 · 50K–200K orders/mo Postgres CPU dominates; product-page tail latency rises Introduce a cache layer for catalog (Redis or KV); horizontally scale API to 4–8 instances. Start sharding hot tables (e.g. webhook_events) by time.
S5 · 200K+ orders/mo Single-region Postgres is the global bottleneck Split read traffic across regional replicas; carve out search into Meilisearch/Typesense; serve images and label PDFs entirely from R2 + CDN. Consider a separate inventory service if SKU count > 50K.

Likely bottlenecks (in order)

  1. Postgres CPU on hot reads — catalog browsing dominates traffic; fix with caching and a read replica.
  2. Stripe webhook handler latency — synchronous DB writes per event. Move to background queue once consistently > 1 RPS.
  3. Inventory contention on hot SKUs — advisory locks serialize checkouts on a single SKU. Becomes visible during flash sales / featured drops.
  4. Object storage upload throughput — only relevant during bulk catalog imports.
  5. Outbound external latency — Shippo rate quotes are ~500–800 ms. Hide with skeleton UI; cache identical rate queries for a short TTL.

API layer scaling

Horizontal scaling — the easy part

The Rust API is stateless modulo the DB pool. Adding instances is a config change on the host. Three things to watch:

  • DB pool sizing. Total connections across instances must stay below the Postgres max_connections. With 2 instances at DATABASE_POOL_SIZE=10 you're at 20; the starter Postgres tier allows ~100. Either raise it as you scale instances, or front with PgBouncer.
  • Migration races on rolling deploys. SeaORM's migration advisory lock prevents two instances from racing; confirmed safe but verify after every migration in staging.
  • In-memory state. Today only the JWKS cache is in-memory. If you ever add idempotency keys, rate-limit counters, or short-lived locks, externalize them (Redis) — don't replicate them in process.

PgBouncer (transaction-pooled) at S3+

Drop a PgBouncer instance between the API and Postgres in session-compatible pooling mode. Avoid transaction-mode pooling for the long-lived Rust API because SeaORM/sqlx rely on prepared statements. Session-compatible pooling lets you scale to many short-lived API instances without exhausting Postgres connection slots. Cost: ~$5/mo for a 256 MB VM; pays for itself the first time you'd otherwise need to bump Postgres a tier just for connections.

Backpressure & rate limiting

tower-http already provides RequestBodyLimit, Timeout, and CatchPanic. Two things to add as load grows:

  • Per-IP rate limit on auth + cart routes (tower-governor) backed by Redis at S3+.
  • Adaptive concurrency limit on the DB pool side — surface 503 before the pool deadlocks.

Database scaling

Vertical first, replicas next, sharding last

  1. Vertical scaling carries you through S3. Going from a 1 vCPU / 2 GB starter tier to 4 vCPU / 16 GB usually 4–8× capacity for 4× the cost. No code change.
  2. Read replica at S3. Two consumers:
    • Admin / BI queries — long-running, must not block the storefront. Point Metabase + admin reports at the replica.
    • Catalog reads — only after caching has been exhausted. Route by feature flag at first to validate consistency.
  3. Sharding is the last resort. webhook_events is the table that grows fastest; partition by month (received_at) once it crosses ~50 M rows. Detach & archive old partitions to S3 in Parquet.

Index hygiene

  • Add pg_stat_statements from day one; review the top 20 by total time monthly.
  • For new indexes after launch, always use CREATE INDEX CONCURRENTLY.
  • Hot indexes today: orders(source, external_order_id), orders(tracking_number) (partial), webhook_events(source, external_id). Don't add more until EXPLAIN says otherwise.

Hot SKU contention

Inventory reservation uses an advisory lock per variant_id. On a featured drop, that variant is a single-threaded bottleneck. Two remedies, in order:

  1. Skip the lock for variants with on_hand > 1000 — most SKUs.
  2. Bucketed counters for ultra-hot SKUs: split a SKU's stock into N buckets, lock a random bucket, fall back to the next on contention. Adds complexity; only do this if you actually see waits.

Assets & CDN

Default at all stages: R2 + Cloudflare CDN

Product images and label PDFs go to Cloudflare R2 with the public bucket fronted by CDN cache. Egress is free, which matters once a product page is being pulled by 10K browsers a day.

S2+ improvements

  • Image optimization at the edge — Cloudflare Images or a Workers image-resize handler for responsive variants. Keeps originals in R2.
  • Aggressive cache headers — Cache-Control: public, max-age=31536000, immutable on hashed asset URLs.
  • Stale-while-revalidate on catalog API responses via Workers KV / Cache API — buys 80% of the benefit of a Redis catalog cache without operating a Redis.

Background work & queues

S1–S2 — inline is fine

Today, Stripe webhook handlers do everything synchronously (insert order, decrement inventory, trigger Shippo label, queue email). For up to a few orders/min this is the simplest correct design.

S3+ — split the webhook

Once webhook bursts cause > 1 s end-to-end on checkout.session.completed, change the shape:

  1. Webhook handler verifies signature, inserts into webhook_events, returns 200 immediately. (Already idempotent.)
  2. Background worker (one extra binary, same crate) reads pending events and processes them.

Two implementation options, in order of preference:

  • Postgres-as-queue (pgqueueing / sqlxmq) — keeps the dependency surface flat, transactional with the rest of the data. Works through S4.
  • Redis Streams via fred or bb8-redis — adopt if you already have Redis for caching/locks at S4+.

Cron-triggered jobs

Marketplace inventory push (Section 6 v2 of the design doc) is the first cron. Run it from Cloudflare Cron Triggers hitting an admin endpoint, or from a tiny cron binary on the API host.

Catalog search

S1–S3 — Postgres full-text is enough

Add a tsvector column to products with a trigger that updates on name / description / collection changes. GIN index on it; rank with ts_rank_cd. Covers ~10K SKUs effortlessly.

S4+ — Meilisearch / Typesense

Once you need typo tolerance, faceting on materials/sizes/price ranges, and "related products" by vector similarity, host a Meilisearch instance (or use Typesense Cloud). Index changes are pushed from a background job triggered by products / product_variants writes. Postgres remains the truth.

Multi-region considerations

The storefront is already multi-region (Workers run on Cloudflare's global edge). The API server is single-region until volume justifies otherwise.

When to go multi-region for the API

  • ≥ 25 % of orders from a different continent than the API region
  • P95 transcontinental latency > 400 ms despite Workers in front
  • Regulatory data-residency request (EU customer data in EU)

What that looks like

  • Primary Postgres in one region, read replicas in others. Writes always go to primary.
  • API instances per region, talking to local replica for reads, primary for writes.
  • Webhooks land in any region but funnel writes to the primary; rely on idempotency.

This is a meaningful step-up in operational complexity. Don't do it on principle — do it when the latency or compliance metric demands it.

Team & tooling scaling

  • Solo / 1 eng — current setup. cargo + npm + Docker; deploys via Wrangler / Fly CLI. Memory > ceremony.
  • 2–3 eng — add a PR template, mandatory CI checks, branch protection, and an on-call rotation (even if it's just two people swapping weeks).
  • 4+ eng — adopt a feature-flag service (Unleash self-host, or GrowthBook). Carve the codebase into clearer ownership boundaries: domain/cart, domain/checkout, etc. — already structured for this.
  • Customer-facing on-call — once you have non-engineer ops, build the admin UI search/filter UX before any new feature. They will save you 10× their own time.
What not to do early. Don't split into microservices; don't adopt Kubernetes; don't write a custom orchestrator; don't move catalog into an "Event Sourcing" system. Each is a credible answer to a real problem you do not have yet, and each will cost more than it returns at S1–S3.
