API Endpoints — Overview

Where REST is used, where GraphQL might be used, and how the documentation is structured.

Stance: REST-first, GraphQL only when proven necessary

Default to REST. All public, cache-heavy, and stable flows — product catalog, cart, checkout, orders, account, webhooks — are REST endpoints under /api/v1/* served by stellaux_server.

GraphQL is not used today. It will only be considered if a future workload (complex dashboard aggregations, mobile app multiplexing) genuinely benefits from it — and even then it would be additive (a /graphql endpoint that aggregates over existing REST), never a replacement.

Where each style lives

SurfaceStyleStatusWhy
Public storefront API (/api/v1/*) REST + JSON Live Cacheable at the CDN, easy to consume from the SSR Worker and the browser, predictable shape.
Webhook receivers REST Live Stripe and Shippo only speak REST; signature is verified on the raw body.
Admin / internal tooling (Phase 3) REST Planned Same server, same conventions, lives on the private VPS. See Internal Tools.
Analytics / dashboard aggregations REST today; candidate for GraphQL in a later phase Future Multiple parallel reads over a single screen would benefit from a single round-trip query.
Background workers (catalog sync, cron) Internal Rust function calls; REST when crossing process boundaries Planned No external clients; in-process is simplest. Workers may call admin REST endpoints with a service token.

Documentation policy (per API Guidelines)

Code-first OpenAPI

Every REST handler is annotated with utoipa attributes directly on the Axum function and on its DTOs (ToSchema, IntoParams). The build fails if annotations are invalid.

Spec is a build artifact

openapi.json is emitted as part of the build (via a build script or a test that writes the file). CI fails on accidental drift against the stored reference.

Interactive docs at /docs

utoipa-swagger-ui (or utoipa-redoc) reads the generated spec and renders an interactive page in every environment. The static HTML you're reading is the architectural summary; the runtime page is the contract.

URL path versioning

Routes live under /api/v1/. Breaking changes promote a new version (/api/v2/) and keep the old version alive for at least one deprecation cycle.

Required annotations

Every endpoint must declare: summary, description, request body schema, response status codes (200, 4xx, 500) with examples, security requirements, and parameter descriptions.

GraphQL deferred

If introduced, async-graphql with built-in introspection (GraphiQL / Playground). It would not be documented via OpenAPI.

Conventions across the REST surface

Base URL

  • Dev: http://localhost:8080
  • Staging: https://api.staging.example.com
  • Production: https://api.example.com

Common headers

HeaderRequired forNotes
Authorization: Bearer <jwt>Protected + admin routesHS256 (first-party) or RS256 (Supabase JWKS) — see Supabase auth.
Content-Type: application/jsonAll POST/PATCHWebhook routes accept the raw provider body; signature verified before parsing.
X-Request-IdOptional, allEchoed back; if omitted, server generates a UUIDv4. Used to correlate logs/traces.
Stripe-SignatureStripe webhook onlyHMAC over raw body using STRIPE_WEBHOOK_SECRET.
X-Shippo-SignatureShippo webhook onlyHMAC-SHA256 with SHIPPO_WEBHOOK_SECRET.

Response shape

JSON on every route. Success responses return the resource directly (no envelope). Errors return:

{
  "error": {
    "code": "INVENTORY_UNAVAILABLE",
    "message": "Variant abc-123 has 0 available units",
    "request_id": "01HVZ7…"
  }
}

Status codes follow standard HTTP semantics. Validation errors return 422 with a per-field detail array.

Pagination

List endpoints accept ?page=1&per_page=20 (defaults 1 / 20, max 100) and return:

{ "items": [ … ], "page": 1, "per_page": 20, "total": 137 }

Idempotency

Webhook handlers are idempotent on (source, external_id) via the webhook_events table. Mutating client endpoints accept an optional Idempotency-Key header — under load this becomes mandatory; see Future Scaling.
