Stripe — payments

Surface area

  • Checkout Sessions — server creates a hosted session, redirects the user.
  • PaymentIntents — implicit, owned by the session.
  • Refunds — issued from the admin UI; logged with actor_user_id.
  • Stripe Tax — optional (automatic_tax: { enabled: true }); 0.5% per order.
  • Webhooks — Stripe → POST /api/v1/webhooks/stripe.

Events we handle

EventAction
checkout.session.completedCreate orders + order_items; decrement inventory; trigger Shippo label purchase; queue order confirmation email.
checkout.session.expiredRelease reserved inventory; mark cart abandoned.
payment_intent.payment_failedRelease reservations; log; no email.
charge.refundedUpdate orders.fulfillment_status = 'refunded'; restock if pre-fulfillment; queue refund email.
charge.dispute.createdNotify admin; do not auto-refund.

Authentication

  • Outbound: STRIPE_SECRET_KEY sent as bearer.
  • Inbound (webhooks): HMAC over the raw body using STRIPE_WEBHOOK_SECRET. Signature verified before any JSON parse.

Idempotency

Stripe assigns each event an id. The handler does INSERT INTO webhook_events(source='stripe', external_id=event.id) ON CONFLICT DO NOTHING and exits early on conflict. orders.stripe_checkout_session_id also has a UNIQUE constraint as a second line of defense. Stripe retries up to 3 days.

Worker compatibility note: the Stripe SDK is not used; we call the REST API via reqwest. For signature verification we compute HMAC-SHA256 ourselves with ring / RustCrypto, so the server stays libc-clean.

Shippo — shipping & tracking

Three points of contact

  1. Rate quote at checkout (synchronous, user-visible)
    POST https://api.goshippo.com/shipments/ — input: from = warehouse env vars, to = customer address, parcels = computed weight + box dims. Output: rates[], filtered to 2–3 service levels (USPS Priority, UPS Ground, expedited).
  2. Label purchase (async, triggered by checkout.session.completed)
    POST /transactions with the chosen rate + label_file_type: 'PDF_4x6'. Stores shippo_transaction_id, tracking_number, tracking_carrier, tracking_url on the order. PDF uploaded to object storage at shipping-labels/{order_id}.pdf.
  3. Tracking webhook → POST /api/v1/webhooks/shippo
    Event of interest: track_updated. Maps Shippo tracking_status.status → our orders.status (PRE_TRANSIT → no change, TRANSIT → shipped, DELIVERED → delivered, RETURNED/FAILURE → admin alert).

Authentication

  • Outbound: SHIPPO_API_TOKEN as bearer.
  • Inbound (webhooks): HMAC-SHA256 with SHIPPO_WEBHOOK_SECRET over raw body; constant-time compare via crypto.subtle equivalent.

Retry policy

Label purchase retries up to 3 times with exponential backoff (1 s, 4 s, 16 s). After 3 failures the order is left fulfillment_status = 'unfulfilled' and an LowStockAdminAlert-style email is queued. Operator can manually retry from the admin UI.

Resend — transactional email

Use

All transactional sends — order confirmation, shipping notification, delivered follow-up, refund confirmation, low-stock admin alert. Templates live in src/emails/*.tsx on the client side and are pre-rendered to HTML at build time, then sent server-side.

Authentication

RESEND_API_KEY as bearer. From-address is configured by RESEND_FROM_EMAIL.

Failure model

Emails are not in the critical checkout path. The webhook handler enqueues the send; a failure is logged and surfaces in the stellaux_external_calls_total{service="resend",success="false"} counter, but the order is still considered successful.

Supabase — Postgres & auth

Primary datastore

Supabase (managed Postgres 16) is the application's system of record for catalog, inventory, carts, orders, customers, webhook events, and the audit log. The Rust API connects via SeaORM over sqlx-postgres + rustls — no libpq, no Supabase client SDK on the server side.

Why Postgres (vs. a document store like Firestore)

  • Relational joins. Catalog ↔ variants ↔ inventory ↔ channel listings is a four-way join we run on every product detail load; SQL is the natural fit.
  • Analytics. The Phase 4 AI analytics agent runs sales-trend queries directly in SQL — no nested-document gymnastics.
  • Transactions. Inventory reservation + order creation must be atomic. Postgres gives us this for free with row-level locks and advisory locks.
  • Portability. Standard Postgres — we can move off Supabase to any managed Postgres provider without code changes.

Supabase-specific features we use

  • Auth — optional JWKS-verified JWT (see next section).
  • Realtime — published on a per-table basis (Phase 3 internal dashboard, see Internal Tools).
  • Row-level security policies — deny-by-default on public business tables unless an explicit policy is added later.
  • PITR + nightly backups — see Deployment → DR.

Connection

DATABASE_URL=postgres://…@db.<project>.supabase.co:5432/postgres
DATABASE_POOL_SIZE=10

The Rust API should use the direct Postgres connection on :5432 or a session-compatible pooler introduced deliberately later. Do not use transaction-mode pooling for SeaORM/sqlx prepared-statement workloads.

Schema exposure posture

  • public is not automatically public data. Supabase PostgREST can see the schema, so every business table there must have explicit RLS posture.
  • private holds internal operational tables. Inventory, reconciliation, and other internal-only records should live outside PostgREST-facing defaults.
  • The Rust API remains the primary business interface. Direct Supabase client access is an explicit exception, not the baseline.

Auth modes

The codebase supports two auth modes:

  • First-party — passwords stored Argon2-hashed in our users table; JWTs signed with JWT_SECRET (HS256), TTL JWT_EXPIRY_SECONDS.
  • Supabase-issued JWT — if SUPABASE_JWKS_URL is set, the server fetches keys from Supabase's JWKS endpoint, caches them in JwksCache (TTL = JWKS_TTL_SECONDS), and validates tokens against the cached key set.

The two modes can coexist: the auth extractor tries the local HS256 key first and falls back to JWKS RS256 verification if a kid is present in the header.

Supabase Storage — product images

Product images are stored in Supabase Storage and accessed through the backend storage contract. Postgres stores image metadata and object keys; Supabase Storage holds the file bytes.

The application contract stays the same regardless of provider details: store only object keys and metadata in Postgres, and resolve public URLs through backend configuration rather than hard-coding vendor paths.

Etsy & eBay — sales channels

Marketplaces are designed for from day one but the connector code is explicitly deferred. The data model carries them:

  • channel_listings(variant_id, channel, external_listing_id, sync_status)
  • orders.source ∈ {website, etsy, ebay}
  • orders.external_order_id for non-website sources
  • inventory_adjustments.channel for channel-attributed deltas

Cloudflare — Workers, DNS, CDN

The storefront deploys to Cloudflare Workers via the @cloudflare/vite-plugin (config in client/wrangler.jsonc). Cloudflare gives us:

  • Workers for SSR — global edge, sub-50 ms TTFB targets.
  • CDN for static assets — automatic.
  • R2 (optional) — object storage with no egress fees.
  • DNS — with proxied records so origin IPs aren't exposed.
  • Cron Triggers (future v2) — drives marketplace sync.

The Rust API does not run on Workers; it runs on a container host (Fly.io / Railway / a VM). Workers proxy auth/cart/checkout calls or the browser hits the API directly via CORS — see Deployment.

Failure modes & recovery

ServiceIf it's downWhat still worksRecovery
Stripe API Checkout button errors; no new orders Browsing, account, admin Stripe outages are rare and short; surface a clear "payments temporarily unavailable" banner; no data action needed.
Stripe webhooks Orders stall in "paid, not recorded" state Customer already has a Stripe receipt Stripe retries up to 3 days; on resolution the handler runs idempotently. Replay manually from the Stripe dashboard if needed.
Shippo rate API Checkout cannot show shipping options Cart, browsing Show fallback flat-rate shipping; allow checkout to proceed; label purchased later.
Shippo label purchase Order is paid but unlabeled Everything else Auto-retry 3× with backoff; admin alert; manual retry button in admin UI.
Resend Customers don't receive emails Orders, payments, fulfillment Email job marked failed in metrics; can be replayed from webhook_events log; no data loss.
Object storage New image upload fails; label PDF not retrievable Catalog browsing (cached CDN), checkout Buy label and store metadata even if PDF upload fails; retry PDF push later.
Postgres Hard outage — full downtime Nothing dynamic This is the single largest dependency; managed-Postgres SLA + PITR + daily logical backup off-site.
