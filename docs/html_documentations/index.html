Architecture Overview

A self-hosted commerce stack: a TanStack Start storefront in front of a Rust (Axum + SeaORM) API, with Postgres as the single source of truth and Stripe, Shippo, and Resend handling payments, shipping, and email. Designed to coexist with Etsy and eBay as parallel sales channels.

Request lifecycle

1 · Edge ingress

Storefront pages render on Cloudflare Workers (TanStack Start SSR). Server functions and API calls from the browser hit the Rust API directly over HTTPS — Workers do not proxy data-plane requests in steady state.

2 · Middleware stack (tower-http)

Every request flows through, in order:

  • SetRequestIdLayer / PropagateRequestIdLayer — UUID per request, echoed on x-request-id.
  • SensitiveRequestHeadersLayer — strips Authorization, Cookie from logs.
  • TraceLayer — span per request, recorded by tracing.
  • PrometheusMetricLayer — HTTP duration histograms, status counters.
  • CorsLayer — allowlisted origins from CORS_ORIGINS.
  • CompressionLayer + DecompressionLayer — gzip on response/request.
  • RequestBodyLimitLayer — normal 1 MB; webhooks get a larger limit.
  • TimeoutLayer — per-route configurable, default ~15 s.
  • CatchPanicLayer — 500 with request id, panic logged with stack.

3 · Route groups

GroupAuthRoutes
publicnonecatalog, craft, auth, healthz, readyz, metrics
webhookssignatureStripe, Shippo (raw body, larger limit)

4 · Database access

SeaORM connection pool sized via DATABASE_POOL_SIZE (default 10). All entities live under src/entity/; domain logic in src/domain/{catalog,cart,checkout,...}.rs. Mutations that affect inventory run inside transactions with row-level locks (SELECT … FOR UPDATE) or Postgres advisory locks for hot SKUs.

5 · External calls

A single reqwest client (rustls, http2, brotli) is built once at boot and shared across Stripe, Shippo, and Resend clients. Each outbound call records latency + success to stellaux_external_call_duration_ms.

Design principles

Supabase Postgres is the truth

Stripe, Shippo, and the marketplaces are authoritative for their own state, but everything is mirrored into Supabase Postgres so the admin UI and audit trail need only one query. We picked Postgres over a document store (Firestore) deliberately — relational joins, SQL-driven analytics, and the Phase 4 AI analytics agent all work better against a real RDBMS.

Idempotency is a table

webhook_events(source, external_id) is a unique key. Every webhook does INSERT … ON CONFLICT DO NOTHING first, processes, then sets processed_at. Replays are free.

One binary, one image

The API ships as a single static Rust binary in a distroless-style image. No Node, no libpq, no runtime DLL hunt.

Boundaries, not layers

Code is organized by domain (catalog, cart, checkout, …) rather than by technical layer. Each module owns its DTOs, handlers, and queries.

Edge-compatible by default

External SDK calls use reqwest with rustls — no Node-only crypto or libpq — so the same code paths can later move into a Worker if useful.

Recurring cost = transaction cost

We pay Stripe per transaction, Shippo per label, Postgres for storage. No per-seat SaaS in the critical path.
