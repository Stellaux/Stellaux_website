Observability

Three pillars — logs, metrics, traces — plus the request-id glue that ties them together. Concretely: tracing for structured logs, axum-prometheus for metrics, opt-in OpenTelemetry for traces.

Stance

Cheap, vendor-neutral, opt-in. The default build emits JSON logs to stdout and Prometheus metrics on /metrics — enough to run a small business. OpenTelemetry trace export is feature-flagged off by default because the OTel SDK pulls ~30 crates and ~60 s of build time; enable it when a collector exists, not before.

Structured logging

Library & format

tracing with the tracing-subscriber JSON formatter (configured in common/bootstrap.rs::init_tracing). Filter controlled by the RUST_LOG env var (e.g. info,stellaux_server=debug,sqlx=warn).

Every log line carries

  • timestamp (RFC 3339, UTC)
  • level (TRACE … ERROR)
  • target (module path)
  • span chain — every request opens a top-level span via TraceLayer
  • request_id — propagated from x-request-id header / generated by SetRequestIdLayer
  • Domain fields the call site attaches (e.g., order_id, variant_id)

What we do not log

  • Authorization + Cookie headers — stripped by SensitiveRequestHeadersLayer.
  • Full shipping addresses, full card numbers — Stripe gives us only last4 + brand, which is what we keep.
  • Webhook payload bodies in steady state — stored separately in webhook_events.payload (DB) for replay, not in logs.

Sample line

{
  "timestamp": "2026-05-28T14:32:08.491Z",
  "level": "INFO",
  "target": "stellaux_server::domain::checkout",
  "span": { "name": "create_checkout_session", "cart_id": "0c4f…" },
  "request_id": "01HVZ7…",
  "fields": { "user_id": "8a2c…", "amount_cents": 12800 },
  "message": "Stripe checkout session created"
}

Shipping logs off-box

The container host streams stdout; from there, any sink works:

  • Fly → built-in log shipper to a Vector/Loki target
  • Railway → Logtail/BetterStack drain
  • Bare host → journald + vector → Loki / Honeycomb / Datadog

Retention: 14 days hot, 90 days cold (cheaper tier). Drop fully after 90 days unless tied to an audit obligation.

Distributed tracing

Wiring is documented in common/opentelemetry.rs as a stub — uncomment the otel feature in Cargo.toml, set OTLP_ENDPOINT, build with cargo build --features otel.

Compatible backends

  • Grafana Tempo (self-host) — fits the Loki + Prometheus + Tempo trio
  • Honeycomb — recommended if buying a SaaS; killer query UI
  • Jaeger — OK for local debugging, less so for prod retention
  • Datadog APM — only if Datadog is already in the stack

What we instrument first

  1. HTTP request span (already created by TraceLayer) — becomes the root
  2. SeaORM query spans — automatic via the debug-print feature plus a custom tracing hook
  3. External calls — span per Stripe / Shippo / Resend call, with service + operation attributes
  4. Webhook handlers — span per event, attributed with source and event_type

Sampling

Head sampling at 10 % in production for happy-path traffic; always sample webhook handlers and any request that returns 5xx. Tail sampling at the collector is a v2 once volume justifies it.

Request IDs & correlation

The x-request-id header is the single string that ties logs, metrics-exemplars, and traces together for a single request.

  1. SetRequestIdLayer generates a UUIDv4 if the inbound request has none.
  2. TraceLayer attaches it to the root span.
  3. Every downstream call (DB, Stripe, Shippo) propagates it as x-request-id on outbound requests.
  4. PropagateRequestIdLayer echoes it on the response.
  5. The storefront surfaces the request id on error screens so customer support can paste it into a log query.
Practical debugging: a customer says "checkout failed". They send the request id from the error toast. grep request_id=… across the log sink, you see the whole story — Postgres query, Stripe call, response code — in under a minute.

Dashboards

Health (single pane of glass)

  • Requests/sec by status class
  • p50 / p95 / p99 latency by route
  • /readyz status across instances
  • Active DB connections
  • Build/image SHA pinned at the top

Commerce

  • Orders/hour by source
  • AOV trend (from order amount histogram)
  • Refunds/day with reason breakdown
  • Inventory units consumed vs restocked
  • Checkout funnel: cart → session → completed

External

  • Stripe / Shippo / Resend call rate & success %
  • p95 outbound latency by service, operation
  • Webhook events received vs processed
  • Failed webhook backlog (processed_at IS NULL AND received_at < now() - '1h')

Auth & abuse

  • Login successes vs failures
  • Signup rate
  • Password reset triggers
  • 5xx by IP (top 10) — abuse indicator

Audit trail

In addition to logs, sensitive admin actions write rows to a separate audit_log table (see common::audit). This is the record we turn to when "who refunded that order?" comes up.

  • actor_user_id, actor_role
  • action — e.g. order.refund, inventory.adjust
  • target_type + target_id
  • metadata (jsonb)
  • request_id — joins back to logs/traces
  • created_at

Retention: indefinite. The audit log is append-only; there is no UI to edit or delete entries.

Frontend observability

  • Cloudflare Web Analytics for traffic / referrers — privacy-friendly, no cookies, no extra script weight.
  • Browser error tracking via Sentry (or any OTel-compatible RUM) — capture unhandled exceptions and surface them with the same x-request-id the API saw.
  • Core Web Vitals sampled via the web-vitals library, sent to a single API endpoint POST /api/v1/internal/web-vitals which records to a histogram.
  • Source maps uploaded on every prod deploy so stack traces are de-minified.
