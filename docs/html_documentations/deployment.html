CI/CD pipeline

Documented under docs/CI/ (filled in alongside the GitHub Actions workflow). High-level stages: 

   commit ──▶ lint ──▶ test ──▶ build images ──▶ deploy staging ──┐
                                                                  ▼
                                                       smoke tests (cypress/k6)
                                                                  │
                                                                  ▼
                                            tag vX.Y.Z ──▶ promote to prod

Per-PR (every push)

  • cargo fmt --check + cargo clippy -- -D warnings
  • cargo test --workspace (DB-backed tests against ephemeral Postgres service)
  • npm run lint, npm run build on the storefront
  • Trivy / cargo-audit scan on dependencies

On merge to main

  • Build + push the Rust API container to GHCR with two tags: main-<sha> and main
  • Build the storefront via wrangler deploy --env staging
  • Deploy API to staging (rolling)
  • Run e2e smoke (login → add to cart → Stripe test checkout → webhook → order row exists)

On tag v*

  • Promote the exact same image SHA to production (no rebuild)
  • wrangler deploy --env production for the storefront
  • Run post-deploy probe against /readyz and /metrics

API server deployment

Container image

Built from stellaux_server/Dockerfile. Multi-stage:

  1. builder: rust:1.84 base, cargo build --release --locked, LTO thin, single codegen unit, symbols stripped.
  2. runtime: distroless or debian:bookworm-slim + CA certs. Binary at /usr/local/bin/stellaux_server. Non-root user. ~30 MB final image.

Why Rust + static binary

  • No libpq — sqlx-postgres + rustls — image stays clean.
  • Startup < 200 ms; suits rolling restarts without warmup pain.
  • Memory footprint < 50 MB idle, fits 256 MB instance comfortably.

Host options (any one)

HostWhy pick itNotes
Fly.io Simplest path; multi-region, free-ish for small loads, manages TLS Native rolling deploys; secrets via fly secrets set.
Railway / Render One-click container deploy; managed Postgres in same project Beware bandwidth pricing if asset egress is heavy — use R2 + CDN.
Hetzner / DO + Caddy Cheapest at small scale; full control You own backups, OS patches, log rotation. Use systemd unit or docker-compose.
Kubernetes (ECS, GKE, …) Only if multi-service infra already exists Overkill for v1; revisit at the scaling thresholds in Future Scaling.

Health gates

  • GET /healthz — liveness; returns 200 if the process is up. No DB check.
  • GET /readyz — readiness; pings DB, fails if pool exhausted or DB unreachable. Load balancer routes traffic only when this returns 200.
  • GET /metrics — Prometheus scrape; not exposed publicly in prod (firewall to monitoring network).

Rolling deploys

  1. New version pulls image, starts, applies migrations on boot (no-op if none pending — sea-orm tracks state in seaql_migrations).
  2. /readyz passes → host adds instance to LB.
  3. Old instance drained: stops accepting new requests, completes in-flight (Axum graceful shutdown on SIGTERM, controlled by shutdown_signal() in server.rs).
  4. Loop until all instances replaced. Two instances minimum so this is non-disruptive.

Storefront deployment

client/ builds via Vite with @cloudflare/vite-plugin and deploys to Cloudflare Workers using Wrangler. Config in client/wrangler.jsonc. 

# local
npm run dev                          # vite dev server, port 3000

# build for Workers
npm run build                        # outputs Worker bundle + static assets

# deploy
npx wrangler deploy --env staging
npx wrangler deploy --env production

Public env vars (VITE_*) are baked into the bundle at build time. Server-side env vars and secrets are configured per-environment in wrangler.jsonc or via wrangler secret put.

API base URL

The browser talks to the API at a CORS-allowlisted public URL (e.g. https://api.example.com). The Worker may also proxy certain routes for first-party cookies / origin parity.

Database migrations

Authored as sea-orm-migration modules under stellaux_server/src/migration/. State is tracked in the seaql_migrations table managed by MigratorTrait.

When migrations run

On every process boot, bootstrap::init calls Migrator::up(&db, None) before axum::serve binds. Pending migrations apply; if none, no-op. Outcome: there is never a moment when the API serves traffic against an older schema than it was compiled for.

Rules of the road

  • Additive only. Adding a column with a default is safe online; renaming or dropping is not. Drops happen as two-deploy migrations (deploy ignore-the-column code, then drop).
  • Index creation uses CONCURRENTLY for tables > 1 M rows.
  • Backfills outside migrations, via a CLI job (src/bin/backfill_*.rs) using batched updates with progress logging.
  • No data deletes in migrations. One-way doors are run manually with eyes on.
  • Idempotency: SeaORM tracks applied set; manual edits to seaql_migrations are forbidden.
Multi-instance gotcha: with N instances rolling, two may momentarily try to migrate at once on boot. SeaORM holds an advisory lock during Migrator::up, so only one wins; the others wait, then see no pending work. Verify on the staging deploy after each migration is added.

Release process

  1. Cut a release branch or merge to main; CI builds + pushes image.
  2. Auto-deploy to staging; smoke tests run.
  3. Soak time — minimum 30 min on staging before promoting. Watch error rate and order success rate on dashboards (see Observability).
  4. Tag vX.Y.Z to trigger prod promotion. Same image, no rebuild.
  5. Post-deploy probe hits /readyz, then a synthetic checkout.session.completed webhook in test mode against staging — confirms the handler path didn't regress.
  6. Announce in the team channel with the image SHA and migration list.

Soft launch sequence (initial production)

  1. Day 0 — deploy backend, internal test orders only (real Stripe live, refunded immediately).
  2. Day 1–3 — friends & family, real shipping to 3–5 addresses.
  3. Day 4 — enable for all visitors.

Rollback

Storefront — wrangler rollback <version-id>. Atomic; takes seconds.

API — re-deploy the previous image tag. Two scenarios:

  • No migration in the new release → straightforward rollback to the previous image SHA. Rolling restart restores it.
  • Additive migration in the new release → safe to roll back code (the older binary ignores the new column). Drop the column later in a planned cleanup migration.
  • Destructive migration (rename/drop) → cannot roll forward in place. This is why drops are two-deploy: the first deploy stops reading the column, the second deploy drops it. The first deploy is rollback-safe; the second isn't and requires a PITR restore.

Database — PITR via the managed Postgres provider (covered next section). Rollbacks of data are last-resort; preferred remediation is a forward-fix migration.

Backups & disaster recovery

Postgres

  • Managed provider's PITR (typically 7–35 days depending on plan) — restore to any timestamp within window.
  • Nightly pg_dump to a separate cloud account / bucket as a defense against provider-account compromise.
  • Weekly restore drill in staging — restore last night's dump into a scratch DB and run SELECT count(*) sanity checks. The drill is the only thing that proves backups work.

Object storage

  • R2 / S3 versioning enabled on the bucket; lifecycle rule removes old versions after 30 days.
  • Cross-region replication for label PDFs (compliance) — not for product images (CDN-cached, regenerable).
