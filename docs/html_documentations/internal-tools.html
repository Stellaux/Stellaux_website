Internal Tools — Catalog Management Phase 3 · planned

Weeks 6–9 of the build plan. All internal tooling stays on a private VPS, completely separate from the public storefront. Same Rust binary, same Supabase Postgres, different route group, different network exposure.

Scope (Phase 3)

TaskWhere it livesWhy
Catalog management (inventory, product metadata, pricing) Internal VPS + Supabase Postgres Relational data with complex joins (variants × inventory × channel listings) and SQL-friendly analytics. Supabase Realtime channels handle live updates per-table.
Transaction logging + low-stock alerts dashboard Internal VPS (React + Rust backend) Keep the internal UI and its logic off the public edge. Same auth surface as the admin role, but bound to an internal hostname.
Live product syncing across sales channels (Etsy, Facebook Marketplace, eBay) Internal VPS background worker Scheduled tasks pull/push inventory data; no serverless needed. Worker reads/writes the same Supabase Postgres.
Automated workflows for order confirmation, shipping label generation, order management Internal VPS + Shippo API Background jobs handle these; Shippo webhooks drive realtime status updates.
Datastore choice. Phase 3 uses Supabase (Postgres), not Firestore. A relational store makes catalog joins, transaction reporting, and the Phase 4 AI analytics agent (SQL-driven sales-trend queries) materially easier. Supabase's real-time engine covers the dashboard live-update use case once tables are explicitly published.

Why one binary, two deployments

  • Public deployment binds :8080 on the public LB; internal-only routes are not registered in that build via a feature flag.
  • Internal deployment runs the same image with FEATURES=internal_tools, exposes only on the VPN-side interface, fronted by Caddy with IP allowlist.
  • Schema and migrations are shared; both servers point at the same Supabase Postgres.

Route base path

All internal endpoints sit under /api/v1/internal/* and the route group requires:

  1. Network: request must arrive on the internal interface (validated by Caddy and asserted again by middleware).
  2. Auth: JWT with role ∈ {admin, ops, fulfillment}.
  3. mTLS (optional, recommended): client cert pinned for service-to-service callers (workers, cron jobs).

Products & variants planned

Most of the public admin product surface (/api/v1/admin/products) is reused. Phase 3 adds the bulk and history operations the operator dashboard needs.

GET/api/v1/internal/products/{product_id}/historyadmin/ops
Description

Append-only audit history for one product: every metadata edit, every variant CRUD, every inventory adjustment, with actor + timestamp.

Response 200
[
  { "ts":"…","actor":"alex@…","action":"variant.create","details":{ "sku":"NP-S" } },
  { "ts":"…","actor":"system","action":"inventory.adjust","details":{ "delta": -1, "reason":"website" } }
]
Implementation

Sourced from the existing audit_log + inventory_adjustments tables, merged in SQL with UNION ALL on a normalized projection.

POST/api/v1/internal/products/{product_id}/duplicateadmin

Create a new draft product cloned from an existing one. New handle derived from the source with a numeric suffix until unique. Variants are duplicated with placeholder SKUs ({old_sku}-COPY).

Pricing rules planned

Centralizes per-variant price overrides, discounts (when introduced), and channel-specific markup.

GET/api/v1/internal/pricing/rulesadmin

List active pricing rules with their scope (global / collection / variant) and effective dates.

POST/api/v1/internal/pricing/rulesadmin
Body
{
  "scope": "variant",
  "variant_id": "uuid",
  "channel": "etsy",
  "markup_pct": 5.0,
  "starts_at": "2026-06-01T00:00:00Z",
  "ends_at": null
}
Behavior

Rules are evaluated in order of specificity (variant > collection > global). Channel sync workers consult this table before pushing prices to Etsy/eBay.

Channel sync — Etsy / eBay / Facebook Marketplace planned

Worker-driven; these endpoints let the dashboard inspect and trigger.

GET/api/v1/internal/channels/{channel}/listingsadmin/ops
Path
channeletsy | ebay | facebook

Lists channel_listings rows for the channel plus the joined variant SKU + product name + current price.

POST/api/v1/internal/channels/{channel}/syncadmin
Body (optional)
{ "variant_ids": ["uuid", "uuid"] }

Omit body to sync everything. Returns a job_id; worker picks it up off the Postgres queue and reports back via POST /api/v1/internal/channels/{channel}/sync/{job_id}/status (worker-internal).

Implementation
  • Reads price + inventory + pricing rules from Supabase Postgres.
  • Calls Etsy updateInventory / eBay reviseInventoryStatus / FB Catalog API POST /catalog/items_batch.
  • Writes outcome to channel_listings.sync_status + a row in webhook_events(source='etsy_sync',…) for replayability.
  • Emits stellaux_external_calls_total{service="etsy",operation="updateInventory"}.

Automated fulfillment workflows planned

POST/api/v1/internal/fulfillment/orders/{order_id}/shipops/fulfillment

Manual trigger of the ship workflow: (1) purchase Shippo label, (2) email customer, (3) flip orders.fulfillment_status = 'fulfilled', (4) push tracking to channel if external order.

Most orders run this workflow automatically on checkout.session.completed; this endpoint is the operator-driven retry.

POST/api/v1/internal/fulfillment/orders/{order_id}/returnops
Body
{ "reason": "customer_request",
  "restock": true,
  "refund_amount_cents": 12800 }

Generates a Shippo return label, emails the customer, and pre-records the inventory restock to apply on scan-in.

GET/api/v1/internal/fulfillment/queueops/fulfillment

Operator dashboard: orders awaiting fulfillment, sorted by promised ship date, with packer notes and label status.

Realtime channels (Supabase) planned

The dashboard subscribes directly to Supabase Realtime over a service-role JWT inside the private network. No new endpoints; these are the published tables.

TableEventsDrives
ordersINSERT, UPDATELive "new order" toast; status changes in the queue view.
inventory_levelsUPDATEStock badges across all open dashboard tabs.
channel_listingsUPDATESync status pill turns red the moment a worker reports an error.
webhook_eventsINSERTLive feed for the integrations debugging panel.

Enabling Realtime for a table

-- migration
alter publication supabase_realtime add table public.orders;
alter publication supabase_realtime add table public.inventory_levels;
-- …

Row-level security still applies — the dashboard uses a service-role JWT signed only on the internal VPS.

Open questions before build

  1. Operator headcount. How many simultaneous dashboard users will Phase 3 have? Affects whether realtime subscriptions need a dedicated read replica or share the primary.
  2. Multi-location inventory. Confirmed needed in Phase 3 or deferred? Adds a locations table + a location_id column to inventory_levels if so.
  3. Facebook Marketplace status. FB's Catalog API is the modern path; Marketplace API itself is approval-gated. Confirm we have access before scoping the connector.
  4. Internal hostname & auth. Cloudflare Tunnel + Access? WireGuard? Tailscale + magic DNS? Affects the network middleware design.
  5. Backups for operator-uploaded assets (bulk import CSVs, supplier receipts). Same R2 bucket with a separate prefix, or dedicated?
